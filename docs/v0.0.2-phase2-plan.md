1. Simplify Journal entry format (remove YAML, minimize AI token consumption)
2. Add `serve` command launchd plist for always-on HTTP ingest
3. Ensure Android retries work reliably when daemon restarts

### User's environment

## Change 1: Journal Format Simplification

### Current format (13 lines of YAML per entry)
```markdown
## ログ - 11:51
### 🎤 Voice Inbox

はい音声テスト

```yaml
voice_inbox:
  source: "android-voice-inbox"
  capture_id: "3ff3dd4a..."
  capture_key: "android-voice-inbox:3ff3dd4a..."
  device_id: "Pixel 8a"
  captured_at: "2026-03-20T02:51:54Z"
  discord_channel_id: ""
  discord_message_id: ""
  discord_author_id: ""
  discord_jump_url: ""
  audio_file: "ingest/2026/03/20/3ff3dd4a....wav"
  whisper_model: "gpt-4o-mini-transcribe"
  processed_at: "2026-03-20T11:57:50+09:00"
`` `
```

### Proposed new format
```markdown
## ログ - 11:51
### 🎤 Voice Inbox

はい音声テスト

_11:51 via Pixel 8a_
```

Key design constraint from user: **AIエージェントのトークン負荷を最小化すること**

### Dedup approach

Options to evaluate:
1. **DB-only dedup** — daemon checks `captures.journal_path` is non-empty → skip. No marker in Journal at all. Zero extra tokens. But if journal file is manually deleted, daemon can't re-append.
2. **Minimal HTML comment** — `<!-- c:src:id -->` (very short). ~10 tokens. Invisible in Obsidian preview. Searchable.
3. **Dual-read** — search for BOTH old YAML marker AND new marker for backward compat during transition.

### Backward compatibility
- Old entries (with YAML) already exist in Journal files
- Need to handle: retry of an old capture that has YAML marker, where new code searches for new marker
- Decision needed: dual-read for one release, or accept that old retries may duplicate (acceptable for personal system?)

### Discord-specific concerns
- Discord entries currently include jump URL in YAML (useful for tracing back to original message)
- Proposed: show "Discord" as device label, drop jump URL from Journal
- Jump URL is still in `messages` table in SQLite for debugging if needed

## Change 2: launchd for `serve`

### Current launchd setup
```
com.kai.voice-inbox.poll     — every 5 min, runs `poll --once`
com.kai.voice-inbox.cleanup  — daily, runs `cleanup`
```

### Proposed addition
```
com.kai.voice-inbox.serve    — long-running, KeepAlive, runs `serve`
```

### Design considerations

The `serve` command is fundamentally different from `poll --once`:
- `poll` is fire-and-forget, runs every 5 minutes
- `serve` is a long-running HTTP server that must stay alive

launchd options for long-running:
- `KeepAlive: true` — restarts on crash, starts on boot
- `ThrottleInterval: 10` — don't restart too aggressively after crash
- No `StartInterval` (not periodic)

### Android retry interaction
- WorkManager retry: exponential backoff (30s base)
- If daemon is down for 30 min, Android retries at: 30s, 1m, 2m, 4m, 8m, 16m (~31 min total)
- After that, WorkManager hits OS-level backoff limits (~1-2 hours between retries)
- With launchd KeepAlive, daemon downtime is typically seconds (crash restart)
- Worst case (MacBook lid closed): daemon resumes when lid opens, Android retries on next backoff cycle

### Questions for Codex
1. Is `KeepAlive: true` sufficient, or should we also use `WatchPaths` or `Sockets`?
2. Should `serve` replace the `poll` plist, or run alongside it? (serve already runs poll internally)
3. Environment variable handling: inline in plist or use EnvironmentVariables dict?
4. Should the serve plist load .env, or should all env vars be in the plist itself?
5. Is the Android WorkManager retry strategy adequate for the "MacBook closed half a day" scenario?

## Change 3: Android retry robustness (if needed)

Potential concern: WorkManager's exponential backoff has OS-imposed limits.
After ~8 retries with exponential backoff, retry interval may exceed 1 hour.
For a MacBook that's closed for half a day, the Android app may need:
- Periodic retry (e.g., every 30 minutes) instead of only exponential
- OR: PeriodicWorkRequest alongside the OneTimeWorkRequest

Decision needed: is current retry adequate, or should we add periodic retry?
